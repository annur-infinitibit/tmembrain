# MemscaleDB Concurrent Indices

Every MemscaleDB index type has a concurrent variant that allows safe, simultaneous access from multiple threads. Concurrent indices use interior mutability (a `&self` interface with locks inside) so they can be shared across threads without wrapping them in your own mutex.

<CardGroup cols={2}>
  <Card title="No External Locking" icon="lock-open" color="#10b981">
    All operations take `&self` instead of `&mut self`. Share one index across all your threads safely.
  </Card>
  <Card title="Concurrent Reads" icon="magnifying-glass">
    Multiple threads can search simultaneously. Read locks allow maximum read parallelism.
  </Card>
  <Card title="Concurrent Writes" icon="pen-to-square">
    Multiple threads can add and remove vectors at the same time. Writes are serialized internally.
  </Card>
  <Card title="Cloneable Handles" icon="copy">
    Clone an index handle to share it across worker threads or pass it over a channel.
  </Card>
</CardGroup>

## Available Concurrent Indices

| Index | Concurrent Variant | Construction | Best For |
|---|---|---|---|
| Flat | `ConcurrentFlatIndex` | `new(dimension)` | Exact search, small datasets |
| HNSW | `ConcurrentHnswIndex` | `new(dimension)` | Graph-based ANN, general purpose |
| IVF | `ConcurrentIvfIndex` | `build(ids, vectors, dimension)` | Large datasets, tunable recall |
| LSH | `ConcurrentLshIndex` | `new(dimension)` | High-dimensional hashing |
| Vamana | `ConcurrentVamanaIndex` | `build(ids, vectors, dimension)` | DiskANN-style graph |

## Quick Start

<CodeGroup>
```python Python
import threading
import uuid
from membrain import ConcurrentHnswIndex

# One shared index for all threads
index = ConcurrentHnswIndex(dimension=1536)

def worker(thread_id):
    vec_id = str(uuid.uuid4())
    embedding = [float(thread_id) / 1536.0] * 1536
    index.add(vec_id, embedding)

# Insert from 8 threads simultaneously
threads = [threading.Thread(target=worker, args=(i,)) for i in range(8)]
for t in threads:
    t.start()
for t in threads:
    t.join()

results = index.search([0.5] * 1536, k=10)
print(f"Found {len(results)} results from {len(index)} vectors")
index.close()
```

```javascript JavaScript
import { ConcurrentHnswIndex } from "membrain";
import { Worker, isMainThread, parentPort, workerData } from "worker_threads";

const index = new ConcurrentHnswIndex(1536);

// Clone the handle for each worker thread
const handle = index.clone();

// Add from main thread
index.add(crypto.randomUUID(), Array(1536).fill(0.5));

// Search concurrently
const results = index.search(Array(1536).fill(0.5), 10);
console.log(`Found ${results.length} results`);

index.close();
```

```rust Rust
use memscaledb::{ConcurrentHnswIndex, ConcurrentVectorIndex, VectorId};
use std::sync::Arc;
use std::thread;

let index = Arc::new(ConcurrentHnswIndex::new(1536));

let mut handles = vec![];
for i in 0..8 {
    let index = Arc::clone(&index);
    handles.push(thread::spawn(move || {
        let id = VectorId::new();
        let vector = vec![i as f32 / 1536.0; 1536];
        index.add_concurrent(id, &vector).expect("add");
    }));
}

for handle in handles {
    handle.join().expect("thread panicked");
}

let results = index.search(&[0.5; 1536], 10).expect("search");
println!("Found {} results from {} vectors", results.len(), index.len());
```
</CodeGroup>

## Concurrency Model

### Reads and Writes

All concurrent indices use `parking_lot::RwLock` internally:

- **Read operations** (`search`) acquire a shared read lock. Multiple threads can hold read locks simultaneously.
- **Write operations** (`add`, `remove`) acquire an exclusive write lock. Other operations wait until the write completes.

This means searches can run in parallel with each other but are briefly paused during inserts.

### Handle Cloning

Each concurrent index can be cloned. All clones share the same underlying data structure -- they are handles to the same index, not copies of it.

<CodeGroup>
```python Python
from membrain import ConcurrentFlatIndex

index = ConcurrentFlatIndex(dimension=128)
index.add("v1", [0.1] * 128)

# clone() returns a handle to the same index
clone = index.clone()
clone.add("v2", [0.2] * 128)

assert len(index) == 2  # Both handles see the new vector
assert len(clone) == 2

clone.close()
index.close()
```

```javascript JavaScript
import { ConcurrentFlatIndex } from "membrain";

const index = new ConcurrentFlatIndex(128);
index.add("v1", Array(128).fill(0.1));

const clone = index.clone();
clone.add("v2", Array(128).fill(0.2));

console.log(index.len());  // 2 -- both handles share the same data
console.log(clone.len());  // 2

clone.close();
index.close();
```

```rust Rust
use memscaledb::{ConcurrentFlatIndex, ConcurrentVectorIndex, VectorId};
use std::sync::Arc;

// Wrap in Arc to share across threads
let index = Arc::new(ConcurrentFlatIndex::new(128));
let clone = Arc::clone(&index);  // Rust uses Arc for sharing

let id1 = VectorId::new();
index.add_concurrent(id1, &[0.1; 128]).expect("add");

let id2 = VectorId::new();
clone.add_concurrent(id2, &[0.2; 128]).expect("add from clone");

assert_eq!(index.len(), 2);  // Both Arc refs see 2 vectors
```
</CodeGroup>

## Build-Style Indices

IVF and Vamana require a bulk training set to build their internal structures. Use the `build()` class method (Python) or static method (JavaScript/Rust):

<CodeGroup>
```python Python
import uuid
from membrain import ConcurrentIvfIndex

ids = [str(uuid.uuid4()) for _ in range(1000)]
vectors = [[float(i % 128) / 128.0] * 128 for i in range(1000)]

# Build once with training data
index = ConcurrentIvfIndex.build(ids, vectors, dimension=128)

# Then add more vectors concurrently
import threading
def add_more():
    index.add(str(uuid.uuid4()), [0.5] * 128)

threads = [threading.Thread(target=add_more) for _ in range(4)]
for t in threads:
    t.start()
for t in threads:
    t.join()

results = index.search([0.5] * 128, k=10)
index.close()
```

```javascript JavaScript
import { ConcurrentIvfIndex } from "membrain";

const ids = Array.from({ length: 1000 }, () => crypto.randomUUID());
const vectors = Array.from({ length: 1000 }, (_, i) =>
  Array(128).fill((i % 128) / 128.0)
);

const index = ConcurrentIvfIndex.build(ids, vectors, 128);

const results = index.search(Array(128).fill(0.5), 10);
console.log(`Found ${results.length} results`);
index.close();
```

```rust Rust
use memscaledb::{ConcurrentIvfIndex, ConcurrentVectorIndex, VectorId};

let ids: Vec<VectorId> = (0..1000).map(|_| VectorId::new()).collect();
let vectors: Vec<Vec<f32>> = (0..1000)
    .map(|i| vec![(i % 128) as f32 / 128.0; 128])
    .collect();
let vector_refs: Vec<&[f32]> = vectors.iter().map(|v| v.as_slice()).collect();

let index = ConcurrentIvfIndex::build(&ids, &vector_refs, 128, Default::default())
    .expect("build");

let results = index.search(&[0.5; 128], 10).expect("search");
println!("Found {} results", results.len());
```
</CodeGroup>

## Configuration

All incremental indices (`ConcurrentFlatIndex`, `ConcurrentHnswIndex`, `ConcurrentLshIndex`) accept an optional config object:

<CodeGroup>
```python Python
from membrain import ConcurrentHnswIndex

index = ConcurrentHnswIndex(dimension=1536, config={
    "m": 16,
    "ef_construction": 200,
    "ef_search": 100,
    "distance_metric": "Cosine",
})
```

```javascript JavaScript
import { ConcurrentHnswIndex } from "membrain";

const index = new ConcurrentHnswIndex(1536, {
  m: 16,
  ef_construction: 200,
  ef_search: 100,
  distance_metric: "Cosine",
});
```

```rust Rust
use memscaledb::{ConcurrentHnswIndex, HnswConfig};

let config = HnswConfig {
    m: 16,
    ef_construction: 200,
    ef_search: 100,
    ..Default::default()
};
let index = ConcurrentHnswIndex::with_config(1536, config);
```
</CodeGroup>

## Async Write-Ahead Log

For persistent concurrent workloads, use `AsyncWriteAheadLog` which batches WAL writes in a background thread and achieves 10-100x higher write throughput compared to per-operation flushing.

```rust Rust
use memscaledb::{AsyncWalConfig, AsyncWriteAheadLog, VectorId};

let config = AsyncWalConfig {
    log_path: "data/membrain.wal".into(),
    checkpoint_dir: "data/checkpoints".into(),
    checkpoint_interval: 10_000,
    max_batch_size: 1000,   // flush after 1000 ops
    flush_interval_ms: 100, // or after 100ms
    ..Default::default()
};

let wal = AsyncWriteAheadLog::open(config).expect("open WAL");

// Share the sender across threads
let sender = wal.sender();

let id = VectorId::new();
sender.log_insert(id, &[0.5; 128]).expect("log insert");

// Explicit durability barrier when needed
wal.flush_sync().expect("flush");

// WAL is flushed and shut down on drop
```

### Durability Trade-offs

| Setting | Throughput | Durability |
|---|---|---|
| `max_batch_size: 1`, `flush_interval_ms: 1` | Low (same as sync WAL) | Highest -- flushed after every op |
| `max_batch_size: 100`, `flush_interval_ms: 50` | Medium | Good -- 50ms data loss window |
| `max_batch_size: 1000`, `flush_interval_ms: 100` | High (10-100x) | Acceptable -- 100ms data loss window |

## Performance

Expected throughput scaling on an 8-core system compared to a single-threaded baseline:

| Operation | 1 thread | 4 threads | 8 threads |
|---|---|---|---|
| Search (reads) | 1x | ~3.4x | ~5-6x |
| Insert (writes) | 1x | ~2.5x | ~4x |
| Mixed (80% read) | 1x | ~3x | ~5x |

Write throughput scales less than reads because writes are serialized through the internal write lock.
