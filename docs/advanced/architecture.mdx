# Membrain Architecture

## Overview

Membrain is a high-performance memory layer designed specifically for Large Language Models (LLMs). It provides semantic search, graph-based relationships, and multi-agent coordination to give LLMs persistent memory, context retention, and knowledge retrieval capabilities. Built with a modular Rust core and FFI bindings for Python and JavaScript.

## LLM Integration Patterns

Membrain integrates seamlessly with LLM applications through three main patterns:

### Pattern 1: RAG (Retrieval Augmented Generation)

```python
# Store document chunks
memory.store_fact("Membrain supports semantic search", 0.9)

# Retrieve relevant context
context = memory.search(user_query, limit=5)

# Inject into LLM prompt
response = llm.chat(context + user_query)
```

### Pattern 2: Conversation Memory

```python
# Store conversation history
memory.store_event("chat", f"User: {user_msg}")
memory.store_event("chat", f"Assistant: {assistant_msg}")

# Retrieve past context
history = memory.search("previous discussion", limit=10)

# Maintain context across sessions
response = llm.chat(history + new_message)
```

### Pattern 3: Agent Memory

```python
# Store agent knowledge and state
memory.store_skill("code_review", "Analyzes code quality")
memory.store_goal("Help user debug application")

# Retrieve relevant capabilities
skills = memory.search("debugging", limit=3)

# LLM agent uses memory to plan actions
action = llm.plan(skills + current_task)
```

## Memory Types for LLMs

<Tabs>
  <Tab title="Episodic Memory">
    Stores conversation history and temporal events for LLMs.

    **Types:**
    - `Event` - Chat messages, user actions, system events
    - `Observation` - Insights about user behavior or preferences
    - `Preference` - User/agent preferences with strength

    **LLM Use Cases:**
    - **Chatbot Memory:** Remember past conversations
    - **Session Context:** Maintain context across chat sessions
    - **User Profiling:** Learn user preferences over time
    - **Interaction History:** Track what topics were discussed

    ```python
    memory.store_event("user_msg", "User asked about Python")
    memory.store_observation("User prefers code examples")
    ```
  </Tab>

  <Tab title="Semantic Memory">
    Stores factual knowledge for RAG and knowledge grounding.

    **Types:**
    - `Fact` - Factual statements with confidence scores
    - `Concept` - Definitions and explanations
    - `Entity` - Named entities with type classification

    **LLM Use Cases:**
    - **RAG Systems:** Store document chunks for retrieval
    - **Knowledge Base:** Domain-specific facts for LLM context
    - **Fact Checking:** Verify LLM responses against stored facts
    - **Context Injection:** Provide grounded information to LLMs

    ```python
    memory.store_fact("GPT-4 has 128k context window", 0.95)
    memory.store_concept("RAG", "Retrieval Augmented Generation")
    ```
  </Tab>

  <Tab title="Procedural Memory">
    Stores LLM agent capabilities and workflows.

    **Types:**
    - `Workflow` - Multi-step procedures the agent can follow
    - `Skill` - LLM agent capabilities and tools
    - `Pattern` - Recognized patterns for decision making

    **LLM Use Cases:**
    - **Agent Capabilities:** Track what the LLM agent can do
    - **Tool Selection:** Choose appropriate tools/skills
    - **Workflow Execution:** Follow stored procedures
    - **Learning:** Store successful patterns for reuse

    ```python
    memory.store_skill("code_review", "Analyzes code quality")
    memory.store_workflow("debug", "1. Reproduce 2. Analyze 3. Fix")
    ```
  </Tab>

  <Tab title="Agent State">
    Stores LLM agent goals and task tracking.

    **Types:**
    - `Goal` - High-level objectives for the LLM agent
    - `Task` - Actionable items and current work

    **LLM Use Cases:**
    - **Goal Tracking:** Remember what the agent is trying to achieve
    - **Task Management:** Track multi-step tasks
    - **Planning:** Help LLM agents plan and prioritize
    - **Progress Monitoring:** Track completion of objectives

    ```python
    memory.store_goal("Help user build a web API")
    memory.store_task("Design database schema")
    ```
  </Tab>
</Tabs>

## Core Components

### Storage Layer

**Location:** `crates/membrain-storage/`

Handles persistence of all memory types.

<AccordionGroup>
  <Accordion title="Features">
    - Key-value storage with memory ID indexing
    - Type-specific serialization
    - Atomic operations
    - Statistics aggregation
  </Accordion>

  <Accordion title="Key Files">
    - `src/lib.rs` - Storage interface
    - Stores memories in a durable backend
  </Accordion>
</AccordionGroup>

### Index Layer

**Location:** `crates/memscaledb/` (standalone vector database) and `crates/membrain-index/` (adapter)

The vector database layer consists of two components:
- **memscaledb**: Standalone vector database with six index types (HNSW, Flat, IVF, LSH, Vamana, Sharded), SIMD acceleration, product quantization, persistence, WAL, distributed mode, and multi-tenancy
- **membrain-index**: Thin adapter that bridges memscaledb to membrain-core types

<AccordionGroup>
  <Accordion title="Index Types">
    - **HNSW** -- Multi-layer navigable graph for dynamic inserts (default)
    - **Flat** -- Brute-force exact search with 100% recall
    - **IVF** -- Inverted file with k-means partitioning
    - **LSH** -- Locality-sensitive hashing for high-dimensional data
    - **Vamana** -- DiskANN-style bounded-degree graph
    - **Sharded** -- Multi-shard HNSW with centroid routing
    - All types have thread-safe concurrent variants
  </Accordion>

  <Accordion title="Features">
    - SIMD-accelerated distance computation (AVX2/SSE2)
    - Five distance metrics (Cosine, Euclidean, DotProduct, Manhattan, Chebyshev)
    - Product quantization for memory-efficient storage
    - LRU query caching with automatic invalidation
    - Atomic observability counters
    - Batch search via Rayon parallelism
    - Write-ahead log (sync and async) for crash recovery
    - Memory-mapped read-only indices
    - Distributed mode with consistent hashing and SWIM gossip
    - Multi-tenant namespace isolation
  </Accordion>
</AccordionGroup>

### Graph Layer

**Location:** Integrated in `membrain-core/src/graph/`

Multi-hop graph traversal with attention mechanism.

**Features:**
- Node embeddings with confidence scores
- Attention-weighted edge formation
- Multi-hop query traversal
- Graph pruning for optimization
- Serializable state (base64)

**Architecture:**
```
Nodes: memory_id + embedding + confidence
Edges: Attention-based (cosine similarity × confidence)
Query: BFS/DFS with attention scoring
```

<Steps>
  <Step title="Start Query">
    Start with query embedding
  </Step>
  <Step title="Find Nearest">
    Find k-nearest nodes (hop 0)
  </Step>
  <Step title="Traverse">
    For each hop, traverse edges weighted by attention
  </Step>
  <Step title="Collect">
    Collect scored nodes across hops
  </Step>
  <Step title="Return Results">
    Return top-k results
  </Step>
</Steps>

### Pipeline Layer

**Location:** `crates/membrain-pipeline/`

Processing pipeline for memory ingestion.

<Steps>
  <Step title="Validation">
    Check input constraints
  </Step>
  <Step title="Deduplication">
    Merge similar memories
  </Step>
  <Step title="Storage">
    Persist to storage layer
  </Step>
  <Step title="Indexing">
    Update search index
  </Step>
  <Step title="Graph Update">
    Add/update graph nodes
  </Step>
</Steps>

### Multi-Agent Layer

**Location:** `crates/membrain-multi-agent/`

Coordination for multi-agent systems.

**Features:**
- Shared knowledge base
- Agent identity tracking
- Cross-agent communication via observations
- Collaborative learning

**Location:** `crates/membrain-ffi/`

Exposes C ABI for language bindings.

### Example Flow

```c
// Create client
void* client = membrain_client_new();

// Store fact
char* out = NULL;
int32_t code = membrain_store_fact(client, "Python is great", 0.9, &out);
if (code == 0) {
    // Parse JSON result
    printf("Result: %s\n", out);
    membrain_string_free(out);
}

// Clean up
membrain_client_free(client);
```

## Data Flow

<Tabs>
  <Tab title="Store Operation (LLM Context)">
    ```
    LLM Application → Language Binding → FFI Layer → Pipeline
      ↓
    Validation → Deduplication → Storage → Index → Graph
      ↓
    StoreResult ← JSON ← FFI ← Language Binding ← LLM App
    ```

    **Example:** Storing LLM conversation
    ```python
    # After LLM generates response
    memory.store_event("assistant_msg", llm_response)
    ```
  </Tab>

  <Tab title="Search Operation (RAG)">
    ```
    LLM App (user query) → Language Binding → FFI Layer → Index
      ↓
    Vector Search → Top-K Results → Storage Lookup
      ↓
    SearchResults ← JSON ← FFI ← Language Binding
      ↓
    Context Injected into LLM Prompt → LLM generates response
    ```

    **Example:** RAG workflow
    ```python
    # 1. User asks question
    query = "How do I use async in Python?"

    # 2. Search memory for relevant context
    context = memory.search(query, limit=5)

    # 3. Inject into LLM
    response = llm.chat(context + query)
    ```
  </Tab>

  <Tab title="Graph Query (Complex Reasoning)">
    ```
    LLM App → Language Binding → FFI Layer → Graph
      ↓
    Multi-hop Traversal → Attention Scoring → Top-K
      ↓
    GraphQueryResult ← JSON ← FFI ← Language Binding
      ↓
    Related Concepts → LLM performs complex reasoning
    ```

    **Example:** Multi-hop knowledge retrieval
    ```python
    # Find related concepts through graph traversal
    result = graph.query(query_embedding, max_hops=3)

    # LLM uses interconnected knowledge
    llm.chat(result.nodes + user_query)
    ```
  </Tab>
</Tabs>

## Thread Safety

<Warning>
  - **MembrainClient**: NOT thread-safe. Create one instance per thread.
  - **MembrainGraph**: NOT thread-safe. Use locks or separate instances.
</Warning>

<Info>
  **Internal Storage**: Uses interior mutability where needed (Mutex/RwLock)
</Info>

## Performance Characteristics

### Storage
- **Insert**: O(1) for storage + O(n) for deduplication check
- **Get**: O(1) lookup by ID
- **Delete**: O(1) removal

### Index
- **Search**: O(log n) to O(n) depending on index structure
- **Update**: O(log n) insertion

### Graph
- **Add Node**: O(n) for edge creation (checks all existing nodes)
- **Query**: O(k × h) where k = branching factor, h = hops
- **Prune**: O(e) where e = number of edges
